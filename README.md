# NS3 Queueing Theory Simulation
### Assignment 3 ŌĆö Network Simulation & Queueing Analysis

---

## Overview

This project uses **NS-3** to simulate queueing theory concepts in a real network topology.
Three clients send UDP traffic through a shared bottleneck link to a server, allowing observation
of how packets queue, wait, get delayed, or get dropped under different network conditions.

The project covers three distinct queueing theory scenarios:

| Scenario | Concept | Key Parameter |
|---|---|---|
| 1 ŌĆö Light Load | Stable queue (under-utilised M/M/1) | rate = 300Kbps, bottleneck = 2Mbps |
| 2 ŌĆö Heavy Congestion | Saturated queue (overloaded M/M/1) | rate = 2Mbps, bottleneck = 1Mbps |
| 3 ŌĆö Discipline Comparison | Priority vs Fair Queuing | qdisc = Prio then FqCodel |

## Running the Experiments

All commands are run from `~/ns-3-dev`. Each run saves its output into its own subfolder
inside `results/` so runs never overwrite each other.

---

### Scenario 1 ŌĆö Light Load: Stable Queue

**Concept:** Arrival rate is well below the bottleneck capacity.
Queueing theory (M/M/1) predicts the queue stays near zero, delay is minimal, and no packets are lost.

```bash
./ns3 run scratch/queueing/mm1-queueing -- \
  --rate=300Kbps \
  --bottleneck=2Mbps \
  --qdisc=Fifo \
  --simTime=30 \
  --tag=scenario1_lightLoad_stable
```

**What each flag does:**

| Flag | Value | Meaning |
|---|---|---|
| `--rate` | 300Kbps | Each client sends at 300 Kbps (900 Kbps total) |
| `--bottleneck` | 2Mbps | Bottleneck link can carry 2 Mbps ŌĆö more than enough |
| `--qdisc` | Fifo | Simple first-in-first-out queue, no prioritisation |
| `--simTime` | 30 | Run for 30 seconds |
| `--tag` | scenario1_lightLoad_stable | Saves results to `results/scenario1_lightLoad_stable/` |

---

### Scenario 2 ŌĆö Heavy Load: Congestion and Packet Loss

**Concept:** Arrival rate far exceeds bottleneck capacity.
Queueing theory predicts the queue fills completely, packets are dropped, delay spikes, and throughput collapses.

```bash
./ns3 run scratch/queueing/mm1-queueing -- \
  --rate=2Mbps \
  --bottleneck=1Mbps \
  --qdisc=Fifo \
  --simTime=30 \
  --tag=scenario2_heavyLoad_congestion
```

**What each flag does:**

| Flag | Value | Meaning |
|---|---|---|
| `--rate` | 2Mbps | Each client sends at 2 Mbps (6 Mbps total) |
| `--bottleneck` | 1Mbps | Bottleneck can only carry 1 Mbps ŌĆö severe overload |
| `--qdisc` | Fifo | FIFO queue ŌĆö no protection, fills and drops blindly |
| `--simTime` | 30 | Run for 30 seconds to observe sustained congestion |
| `--tag` | scenario2_heavyLoad_congestion | Saves to `results/scenario2_heavyLoad_congestion/` |

---

### Scenario 3a ŌĆö Queue Discipline: Priority Scheduling

**Concept:** Moderate load (arrival Ōēł capacity). Shows how the **PrioQueueDisc** scheduler
serves smaller packets first, giving lower delay to Client 0 (512B) vs Client 2 (2048B).

```bash
./ns3 run scratch/queueing/mm1-queueing -- \
  --rate=1Mbps \
  --bottleneck=1Mbps \
  --qdisc=Prio \
  --pkt0=512 \
  --pkt1=1024 \
  --pkt2=2048 \
  --simTime=30 \
  --tag=scenario3_Prio
```

**What each flag does:**

| Flag | Value | Meaning |
|---|---|---|
| `--rate` | 1Mbps | Each client sends at 1 Mbps (3 Mbps total) |
| `--bottleneck` | 1Mbps | Bottleneck at 1 Mbps ŌĆö moderate pressure |
| `--qdisc` | Prio | Priority queue: smaller/higher-priority packets go first |
| `--pkt0/1/2` | 512/1024/2048 | Different packet sizes per client to make priority visible |
| `--simTime` | 30 | 30 seconds |
| `--tag` | scenario3_Prio | Saves to `results/scenario3_Prio/` |

---

### Scenario 3b ŌĆö Queue Discipline: Fair Queuing with AQM

**Concept:** Same load as Scenario 3a but using **FqCoDelQueueDisc**.
FqCoDel gives each flow its own queue (fairness) and actively manages delay (AQM).
Compare results directly with Scenario 3a to see the discipline effect.

```bash
./ns3 run scratch/queueing/mm1-queueing -- \
  --rate=1Mbps \
  --bottleneck=1Mbps \
  --qdisc=FqCodel \
  --pkt0=512 \
  --pkt1=1024 \
  --pkt2=2048 \
  --simTime=30 \
  --tag=scenario3_FqCodel
```

**What each flag does:**

| Flag | Value | Meaning |
|---|---|---|
| `--rate` | 1Mbps | Same as 3a ŌĆö identical load for fair comparison |
| `--bottleneck` | 1Mbps | Same bottleneck |
| `--qdisc` | FqCodel | Fair Queuing + CoDel AQM ŌĆö reduces bufferbloat |
| `--pkt0/1/2` | 512/1024/2048 | Same packet sizes as 3a |
| `--simTime` | 30 | 30 seconds |
| `--tag` | scenario3_FqCodel | Saves to `results/scenario3_FqCodel/` |

## All Available Simulation Parameters

| Parameter | Default | Description |
|---|---|---|
| `--simTime` | 15 | Total simulation duration in seconds |
| `--rate` | 500Kbps | Data rate for each client OnOff application |
| `--bottleneck` | 1Mbps | Bottleneck link capacity (router ŌåÆ server) |

| `--accessDelay` | 2ms | Client access link one-way delay |

| `--qdisc` | Prio | Queue discipline: `Prio`, `FqCodel`, `TBF`, `Fifo` |
| `--pkt0` | 512 | Packet size for Client 0 in bytes |
| `--pkt1` | 1024 | Packet size for Client 1 in bytes |
| `--pkt2` | 2048 | Packet size for Client 2 in bytes |
| `--tag` | auto | Name for this experiment run (used as output folder name) |

If `--tag` is not provided, an automatic tag is generated from the parameters,
e.g. `rate500Kbps_bn1Mbps_Prio`.

## Network Topology

```
Client 0 (512B)  ŌöĆŌöÉ
                   Ōöé  100Mbps/2ms     1Mbps/10ms
Client 1 (1024B) ŌöĆŌöĆŌöżŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌ¢║ Router ŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌ¢║ Server
                   Ōöé   access links   [queue disc]   bottleneck
Client 2 (2048B) ŌöĆŌöś
```

- Clients send UDP traffic using an OnOff application (alternating on/off bursts)
- The router applies the selected queue discipline on the bottleneck link
- All three flows share the single 1Mbps (default) bottleneck ŌĆö this is where queuing occurs
- The FlowMonitor measures per-flow statistics end-to-end

## Troubleshooting

**Clock skew warning during build:**
```
gmake[3]: warning: Clock skew detected.
```
This is harmless. Fix it permanently with:
```bash
sudo hwclock --hctosys
```

**`FigureCanvasAgg is non-interactive` warning from Python:**
This is expected in headless/SSH terminals. The scripts use the `Agg` backend which
writes directly to PNG ŌĆö no display server needed. The PNG files are saved correctly.

**`lxml` not found:**
```bash
pip install lxml --break-system-packages
```

**No experiment subfolders found:**
Make sure you ran the NS3 simulation first and that the `results/` folder is inside
`~/ns-3-dev/`, not somewhere else.

**Build fails with `MobilityHelper` not declared:**
Make sure your `mm1-queueing.cc` has this include at the top:
```cpp
#include "ns3/mobility-module.h"
```
